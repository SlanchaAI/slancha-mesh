## The Spark situation, honestly

**Hardware**: NVIDIA GB10 / aarch64 / Linux 6.17 / driver 580.142 / CUDA 13 / 128 GB unified memory.

**vLLM**: 0.17.0 already installed at `~/.local/lib/python3.12/site-packages` (system Python 3.12). `torch 2.10.0+cu128`, `arch_list = ['sm_80','sm_90','sm_100','sm_120']`, but GB10 reports `cuda_capability=12.1` (sm_121). vLLM emits a UserWarning saying "max supported is sm_120; will queue JIT" and keeps going — that's *not* the blocker; **the FP8 GEMM kernel coverage on Blackwell is**.

**The kernel gap and the fix**:

1. Default FP8 path → vLLM picks `Triton` FP8 MoE backend, model loads
   fully (29.1 GiB), then `profile_run` fires `torch.ops._C.cutlass_scaled_mm` →
   `RuntimeError: Error Internal`. No sm_121 cutlass kernel.
2. With `VLLM_USE_FLASHINFER_MOE_FP8=1` → vLLM raises
   `NotImplementedError: no FlashInfer FP8 MoE backend supports the
   configuration` *before* model load. FlashInfer doesn't have GB10
   coverage either.
3. **With `VLLM_TEST_FORCE_FP8_MARLIN=1` → works.** Marlin treats the
   FP8 weights as weight-only quantization (dequant at runtime to BF16,
   then BF16 matmul). vLLM logs:
   ```
   WARNING marlin_utils_fp8.py:97 Your GPU does not have native support for
     FP8 computation but FP8 quantization is being used. Weight-only FP8
     compression will be used leveraging the Marlin kernel. This may
     degrade performance for compute-heavy workloads.
   ```
   "May degrade performance" = we lose the FP8 math speedup, but the
   weights still fit in FP8 memory footprint. Net: 30 GB weights load,
   the matmul runs in BF16, the model serves.

**That's the one-line fix we missed in the initial brief**: set
`VLLM_TEST_FORCE_FP8_MARLIN=1` before `vllm serve`. The flag is named
`_TEST_` but is the documented workaround for "platform without native
FP8 ops."

## Recommendations for v0.0.3

1. **Bake `VLLM_TEST_FORCE_FP8_MARLIN=1` into the bring-up script and
   `VLLMBackend.start()` env.** The flag is currently hardcoded in
   `bring-up-spark.sh`; lift it into `mesh/backends.py:VLLMBackend.start`
   so any FP8 specialist served via vLLM gets the Marlin fallback
   automatically on Blackwell. Detect chip family in the probe and
   inject conditionally.
2. **Add an `engine_test_pass` step to bring-up.** Today the script
   waits for `/health` and runs a smoke chat. Promote that smoke into
   the daemon's `wait_ready()` — confirm an actual forward pass
   produces tokens, not just that the HTTP server is bound. Catches the
   profile_run-succeeds-but-runtime-fails class.
3. **Wire up llama.cpp as the second backend.** Even though vLLM works
   today, GGUF support broadens the catalog and gives us a path for
   smaller (Mac mini, Framework Desktop) nodes. The `BaseBackend`
   Protocol is ready; ~80 LOC of `LlamaCppBackend`.
4. **Cache the BF16 sibling for the same Qwen3.5-27B.** Qwen3.5-27B
   BF16 is already cached (52 GB) — adds a non-FP8 fallback if
   Marlin breaks on a future vLLM release. Add a second TOML card.
5. **Persist the registry's event log to disk.** Today everything is
   in-memory. Spec §2 examples imply event sourcing; v0.0.3 should
   spill events to a single JSONL with fsync on heartbeat. Lets us
   restart slancha-api without losing the cluster state.
6. **Streaming heartbeat from /metrics.** `_parse_vllm_metrics` works
   on a snapshot; for hot-interactive routing we want a 1 Hz pull with
   exponential backoff on transient failures. Currently single-shot.
7. **Multi-specialist on one Spark.** GB10 has 64 GiB free even with
   the 30B MoE resident; we could co-host a 7B math model on
   port 8002. Allocator already supports `full_set` strategy; just
   needs `ServeDaemon` to spawn multiple `VLLMBackend` instances.
8. **Idle fine-tune daemon stub.** Spec §7. Detect <10% util for 60 s,
   set `health="training"`, run a no-op LoRA pass, yield on incoming
   traffic. Don't actually train yet — just wire the lifecycle.

## Bug list for v0.0.3

1. ~~`mesh/backends.py:VLLMBackend.start()` does not set
   `VLLM_TEST_FORCE_FP8_MARLIN`.~~ **Fixed in this session.** The env
   var is now set via `env.setdefault` so any pure-Python launch via
   `ServeDaemon` works on Blackwell. v0.0.3 should make this
   conditional on probe chip family so we don't gimp FP8 on Hopper.
2. `mesh/serve.py:build_daemon` constructs a fresh probe on each call;
   should cache for daemon lifetime.
3. `mesh/backends.py:_find_pid_on_port` uses `ss -ltnp sport = :PORT`
   syntax that varies between distros (works on Spark Linux; Mac mini
   needs `lsof`). Add a `psutil`-based fallback.
4. Heartbeat loop swallows backend errors but doesn't surface them via
   `health="degraded"` immediately — one-tick delay until next
   `is_alive()` poll. Acceptable for v0.0.2; consider a process-side
   callback.
5. `torch.cuda.get_arch_list()` reports `sm_120` max while GB10 is
   sm_121 — every CUDA op goes through PTX JIT, which is ~2-3× slower
   than native kernels. Pin torch 2.11+ when it lands sm_121.
6. vLLM's `vllm:num_preemptions_created` is a tracking gauge with a
   bogus Unix-timestamp value (1.7e9) — not actual count. Our
   `_parse_vllm_metrics` correctly ignores it; document for future
   readers.
7. `mesh/serve.py:_signal_handler` calls `sys.exit(0)` from inside the
   handler — fine for a daemon but bad if `ServeDaemon` is embedded.
   Make the exit conditional on "running as `__main__`".

## Reproducing

```bash
cd ~/Source/slancha-test

# One-shot end-to-end (assumes Qwen3-Coder-30B-FP8 in HF cache)
TORCH_CUDA_ARCH_LIST=12.0 VLLM_TEST_FORCE_FP8_MARLIN=1 \
  vllm serve Qwen/Qwen3-Coder-30B-A3B-Instruct-FP8 \
  --host 127.0.0.1 --port 8001 \
  --served-model-name qwen3-coder-30b-a3b-fp8 \
  --max-model-len 4096 --gpu-memory-utilization 0.55 \
  --dtype auto --trust-remote-code --enforce-eager &

# Wait for /health (4-5 min cold)
until curl -sf http://127.0.0.1:8001/health; do sleep 5; done

# Run unit tests + live integration
VLLM_LIVE_URL=http://127.0.0.1:8001 uv run pytest mesh/tests/ -v
# Expect: 56 passed (52 unit + 4 live).
```

Or use the script:

```bash
bash mesh/scripts/bring-up-spark.sh qwen3-coder-30b-a3b-fp8 8001
```
