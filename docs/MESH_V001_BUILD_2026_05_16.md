# Slancha-Mesh v0.0.1 build — 2026-05-16

**Status**: ready for v0.0.2 (vLLM provisioning + FastAPI server mount).

## What shipped

Implemented the v0.0.1 minimal viable mesh per `docs/SLANCHA_MESH_V0_SPEC.md`
in 7 new Python modules under `mesh/`: hardware probe with GB10
unified-memory detection, 5 specialist TOML cards (math, code, general,
multilingual, reasoning), a pure-function allocator with all three
coverage strategies, an event-sourced in-memory registry with
deterministic snapshot replay, a mesh router that returns
`(specialist, node, queue, fallback_chain)`, and 38 pytest tests
(all passing in 0.24s — single CI shard). The probe was verified live on
the local Spark (GB10 aarch64, 121.6 GB unified RAM, 1.28 TB free disk),
correctly detecting vLLM + Ollama backends and recording the
`nvidia-smi [N/A] VRAM` quirk as a warning instead of crashing. Files +
test count: 14 source files (incl. 5 TOML cards) + 6 test files;
38 tests across 6 modules.

## Insight that should feed the Stripe pitch

**The GB10 lies about VRAM.** `nvidia-smi --query-gpu=memory.total`
returns `[N/A]` on driver 580.142 because GB10 is a unified-memory
architecture — there is no discrete VRAM to report. The spec §3.2 hard
filter `if spec.min_vram_gb > node.vram_available_gb` would have
silently rejected every specialist from every Spark because
`vram_available_gb=None`. Our fix: detect the `[N/A]` string, set
`unified_memory=True`, and fall back to `ram_available_gb - 8GB OS
reserve` as the effective fit budget. Pitch implication: **the
heterogeneous-routing story isn't "different hardware, different
specialists" — it's "different hardware exposes capacity through
completely different APIs."** A naïve placement algorithm that assumes
`nvidia-smi memory.free` is canonical will refuse to host anything on
the most modern consumer Blackwell silicon. The mesh placement layer is
the abstraction that absorbs this. For Stripe: any "schedule workload
onto user hardware" pitch needs to assume the hardware-introspection
layer is broken in vendor-specific ways. We solved this in the probe by
giving every field a `probe_warnings` audit trail.

## Gaps for v0.0.2

1. **Active bandwidth/RTT probe** — `POST /mesh/v1/probe-network` schema
   defined but `probe_node` records `bandwidth_to_master_mbps=None`.
   Needs an iperf-equivalent on boot. Cheap fix.
2. **FastAPI server** — `mesh/registry.py` exposes
   `HeartbeatPostRequest/Response`, `RegistryGetResponse` and a
   `MeshRegistry` class with all the logic. Need ~30 LOC on slancha-api
   to mount the routes. Trivial.
3. **vLLM provisioning script** — `bring-up-mesh-node.sh` per spec §12
   day-5. Not started; the catalog declares `required_backend = "vllm"`
   but nothing wires a node to actually load weights.
4. **Real model downloads** — catalog is pure metadata; no
   `huggingface-cli` invocation. Needs ~20 LOC to read the card →
   download → mark `LoadedModel` in heartbeat.
5. **Backend abstraction** — `Backend` is a Literal enum; no shared
   `BaseBackend` interface yet. v0.0.2 should add a thin protocol so
   nodes can switch between vLLM/llama.cpp without router changes.
6. **Difficulty fall-through ordering** — current router falls easy→
   medium→hard, but for unranked routes between tiers, we lose the
   pareto cost weighting. Should re-use slancha-api's `_pareto_score`
   when the API is reachable from the mesh package (currently
   duplicated to keep mesh standalone).

## Bug found mid-implementation

The router's `_domain_for_signals` originally mapped `"code"` to
`"general"` because the synonym set was `{"computer science",
"engineering", "programming", "coding"}` — missing the canonical short
form. Caught by the e2e test, which routed 2 of 5 prompts to cloud
instead of to spark-2's coder specialist. Fix: add `"code"` to the
synonym set. Test now asserts both Sparks are exercised when math+code
are loaded. **Lesson for spec interpretation**: catalog `domain` values
are the source of truth; the router synonym table must echo them
verbatim before adding aliases.
